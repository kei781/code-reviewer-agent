# Implementation Notes

## SDD Intent

The current implementation follows the ADR/PRD v5 review-server scope. The product prepares a local PR checkout, runs independent Claude Code and Codex reviewer passes, cross-validates findings against the codebase, posts valid review comments, and then hands the decision back to a human maintainer.

## Active Boundary

- GitHub sends PR events to the self-hosted review server.
- The review server prepares the local repository state with clone/fetch/head-SHA checkout.
- The review server launches Claude Code as the local OAuth-authenticated orchestrator.
- Claude Code orchestrates two independent reviewer passes: Claude Code directly and Codex through pre-connected plugin/tooling.
- Candidate findings must be corroborated and must contain local checkout evidence before publication.
- Review output is a signal, not formal approval or merge authorization.
- After review comments are posted, humans decide whether to resolve comments or request additional development.

## Runtime Gap and Next Phase

The implemented source currently provides reusable app/domain/orchestration modules and runtime config, but it does not yet provide a process entrypoint that opens a port. There is no `npm start`, pm2 process file, `GET /healthz`, or `POST /webhooks/github` implementation yet.

The next planned phase is Phase 3, documented in `docs/superpowers/specs/2026-06-09-self-hosted-webhook-server-runtime-design.md`. Phase 3 will add a pm2-runnable self-hosted webhook server runtime in separately reviewable implementation PRs.

## Source Decisions

- Root `ADR.md` and `PRD.md` v5 make Codex the second reviewer, not an implementer role.
- Runtime config is loaded through `src/shared/config.ts`; other TypeScript modules should consume typed config instead of reading `process.env` directly.
- Runtime config uses the local OAuth-authenticated Claude Code command (`claude`) instead of model API keys.
- Codex is a setup prerequisite connected to Claude Code plugin/tooling, not a direct server-side runtime command.
- Setup uses `scripts/setup.sh` for POSIX first-run bootstrap and `scripts/setup.mjs` for dependency/setup verification.
- Logging routes through `log()` so later log sinks can be changed in one shared module.
- `dist/` is cleaned before builds so removed source and test modules cannot leave stale compiled output behind.

## Implemented Files

- `src/shared/log.ts`: central replaceable logging helper.
- `src/shared/config.ts`: single runtime config module for environment-derived values.
- `src/app/runEnsembleReview.ts`: port-driven webhook review use case with eligibility checks, state claim, workspace preparation, independent review orchestration, cross-validation, fingerprint dedupe, and publication.
- `src/app/respondToReviewerMention.ts`: explicit reviewer mention use case with analysis-only response boundaries.
- `src/domain/review/*`: pure review signal, marker, context, and cross-validation contracts.
- `src/domain/policy/*`: pure PR eligibility, risky path, and reviewer trigger policies.
- `src/agents/*`: Claude Code orchestrator, Claude Code reviewer, Codex reviewer, and same-level harness builders.
- `src/orchestration/reviewServerPipeline.ts`: side-effect-free review-server run-plan builder.
- `src/project/*`: directory ownership and phase metadata derived from ADR/PRD.
- `.env.example`: review-server runtime config template for the local OAuth-authenticated Claude Code orchestrator.
- `scripts/setup.sh`: macOS/Linux shell bootstrap that prepares pinned Node/npm with Volta when available or a local `.tools/node` install otherwise.
- `scripts/setup.mjs`: setup automation for `.env`, dependencies, build, and tests.
- `docs/superpowers/specs/2026-06-09-self-hosted-webhook-server-runtime-design.md`: planned Phase 3 runtime design for a pm2-runnable webhook server.

## Removed Stale Scope

Earlier implementation notes and code had carried forward future automation concepts. Those are no longer active runtime scope:

- No implementer role runtime config.
- No model API key requirement for Claude Code or Codex reviewer execution.
- No label-driven automatic code modification.
- No patch generation/apply loop.
- No loop terminal-state contract.
- No branch-protection verdict check or merge automation.
- No low-risk autonomous readiness policy.

## Red-Green Notes

- A failing phase-scope test first proved that stale runtime modules and exported contracts still existed.
- The green implementation removed those modules and reduced phase metadata to Phase 0 through Phase 2.
- Review publication metadata was simplified from loop-oriented state to review-only publication state.
- Existing config tests preserve the single environment-read boundary and verify stale runtime env keys are absent.

## Human Editability Rules

- Each source file has one clear reason to exist.
- Domain policy remains pure and does not import adapters.
- App use cases coordinate through explicit ports.
- Harness builders return plain strings and can be edited without understanding SDK clients.
- Runtime execution is represented through ports or run-plan data until concrete adapters own side effects.

## Setup Usage

- First run on macOS/Linux: `sh scripts/setup.sh`.
- Existing Node 24+ environments: `npm run setup`.
- The shell bootstrap prefers Volta when available. Without Volta, it installs Node.js under `.tools/node` and does not require `sudo` or a global Node install.
