# Implementation Notes

## SDD Intent

The current implementation follows the ADR/PRD v5 review-server scope. The product prepares a local PR checkout, runs independent Claude Code and Codex reviewer passes, cross-validates findings against the codebase, posts valid review comments, and then hands the decision back to a human maintainer.

## Active Boundary

- GitHub sends PR events to the self-hosted review server.
- The review server prepares the local repository state with clone/fetch/head-SHA checkout.
- The review server launches Claude Code as the local OAuth-authenticated orchestrator.
- Claude Code orchestrates two independent reviewer passes: Claude Code directly and Codex through pre-connected plugin/tooling.
- Candidate findings must be corroborated and must contain local checkout evidence before publication.
- Review output is a signal, not formal approval or merge authorization.
- After review comments are posted, humans decide whether to resolve comments or request additional development.

## Phase 3A Runtime

The implemented source now provides a Phase 3A process entrypoint that opens a Node.js HTTP server. It includes `npm start`, `npm run serve`, a pm2 process file, `GET /healthz`, `POST /request_reviewer/webhook`, and the legacy-compatible `POST /webhooks/github`.

The Phase 3A webhook route captures a bounded raw body, aborts oversized streams as soon as the configured limit is crossed, verifies `X-Hub-Signature-256` before JSON parsing, applies optional repo allowlist checks, recognizes supported `pull_request` and `issue_comment` events, logs through `log()`, and returns safe `202` acknowledgements. GitHub publication, git workspace preparation, persistent state, and Claude Code orchestration now live behind Phase 3B and Phase 3C adapters. Phase 3D wires recognized webhook deliveries into the app use cases through an explicit dispatcher while keeping long-running review execution off the HTTP acknowledgement path.

The runtime guide exposed at `GET /request_reviewer/webhook` now maps GitHub's webhook UI directly: choose `Let me select individual events.`, leave `Just the push event.` unselected, subscribe only to Pull requests plus Issue comments, and do not select the similarly named Pull request reviews/comments/threads events.

## Phase 3B Adapters

The implemented source now provides the Phase 3B adapter layer for GitHub payload mapping, server-side GitHub App token exchange, GitHub REST calls, review publication, git workspace preparation, and SQLite state persistence.

The GitHub mapper reads repository identity, clone URL, pull request number, branches, head SHA, draft/closed/fork state, labels, and comment metadata from webhook payloads plus injected API metadata where GitHub payloads are incomplete. It does not use static `GITHUB_OWNER` or `GITHUB_REPO` routing. Pull request mapping fails closed when changed paths are unavailable, and `runEnsembleReview` also rejects empty changed path lists before claiming work.

The workspace adapter prepares a human-readable checkout path under `REVIEW_SERVER_WORKSPACE_ROOT`, runs `git clone --no-checkout` or updates an existing checkout, fetches the webhook head branch, and checks out the exact webhook head SHA in detached mode. Shell execution is isolated behind a `CommandRunner` interface.

Review hardening now rejects git positional inputs that could be parsed as options: repository URLs and head branches must not start with `-`, and the head SHA must be a 40-character hexadecimal Git object id before any git command runs. Git commands also run with per-command isolation: hooks point to a controlled empty hooks directory, fsmonitor is disabled, terminal prompts are disabled, system git config is ignored, and global git config points to a controlled empty file under the workspace root.

The SQLite state store keeps review claims, posted finding fingerprints, review failures, follow-up claims, follow-up responses, and follow-up failures behind app ports. Duplicate webhook deliveries, duplicate PR head-SHA review work, and duplicate follow-up comment revisions are skipped across process restarts.

The GitHub publisher obtains server-side installation tokens through a GitHub App JWT provider, posts review comments with `COMMENT` events only, publishes skip/failure/follow-up issue comments, and never exposes approval, merge, or code modification behavior.

The GitHub REST client uses Node `fetch` behind the adapter boundary for repository installation lookup, installation token creation, pull request review comments, issue comments, and label posting. Failed GitHub API responses throw sanitized errors that do not include bearer tokens.

## Phase 3C Claude Code Orchestrator Adapter

The implemented source now provides the Phase 3C Claude Code orchestration adapter boundary. The adapter launches the locally installed, OAuth-authenticated Claude Code command behind the existing `ReviewOrchestratorPort`; Codex remains a reviewer invoked by Claude Code through pre-connected plugin/tooling, not a server-side Codex command.

The command runner now supports `envMode: "replace"` so agent processes can receive a deliberately constructed environment instead of inheriting the server process environment. `src/adapters/orchestrator/agentEnvironment.ts` keeps local CLI/runtime path variables needed for Claude Code and removes server-side GitHub credentials, webhook secrets, private-key paths, database paths, and review-server config values before launching the agent.

`src/adapters/network/modelEgressGuard.ts` defines the fail-closed egress guard boundary. An empty model allowlist fails before launch, enforcer failures prevent the agent process from starting, and successful sessions provide explicit environment additions plus a disposable policy session. The concrete deployment mechanism remains injectable so local firewall, container, or sandbox policy can be selected outside the reusable domain/app code.

`src/adapters/orchestrator/claudeCodeOrchestratorAdapter.ts` wraps Claude Code execution with the egress guard, passes the orchestrator harness as the command prompt, uses replace-env mode, applies a timeout through the command runner, parses a structured JSON review result between `AI_REVIEW_RESULT_JSON_START` and `AI_REVIEW_RESULT_JSON_END`, validates the finding schema, and sanitizes command failure messages before throwing. The adapter does not default the agent process working directory to the PR-controlled checkout; the checkout path is passed as data in the harness instead. The orchestrator harness now returns server-consumed JSON instead of posting GitHub comments from the agent session.

The app test suite now explicitly covers independent review execution failure. If Claude Code orchestration throws, `runEnsembleReview` publishes a structured failure at `run-independent-reviews`, records the failure after a successful claim, and does not publish review comments.

## Phase 3D Runtime Dispatch Wiring

The implemented source now provides the Phase 3D runtime dispatch boundary. `src/server/httpServer.ts` forwards both recognized webhook metadata and the parsed payload to the runtime callback, then returns `202` without waiting for long-running dispatch work. Callback failures are logged through `log()` instead of turning accepted GitHub deliveries into delayed HTTP responses.

`src/server/webhookDispatcher.ts` routes `pull_request` deliveries through `runEnsembleReview` and reviewer `issue_comment` deliveries through `respondToReviewerMention`. It fetches PR changed paths before pull request mapping and PR head/fork/closed metadata before reviewer mention mapping. Ordinary PR comments without a reviewer trigger skip before metadata reads, app use-case calls, or publication side effects. Mapping failures return structured skip results and do not enter workspace, model, state-claim, or publication side effects.

`src/adapters/github/githubRestClient.ts` now includes read-only PR file and PR metadata methods, including changed-file pagination. `src/adapters/github/githubPullRequestMetadataProvider.ts` obtains installation tokens and keeps token handling out of the dispatcher.

`src/adapters/orchestrator/claudeCodeFollowUpResponderAdapter.ts` implements `FollowUpResponderPort` for reviewer mention responses. It uses the same guarded command execution pattern as the orchestrator adapter, scrubs server secrets from the agent environment, requires JSON between `AI_FOLLOW_UP_RESPONSE_JSON_START` and `AI_FOLLOW_UP_RESPONSE_JSON_END`, rejects non-`analysis-only` response scopes, and rejects stale `reviewedSha` values.

## Source Decisions

- Root `ADR.md` and `PRD.md` v5 make Codex the second reviewer, not an implementer role.
- Runtime config is loaded through `src/shared/config.ts`; other TypeScript modules should consume typed config instead of reading `process.env` directly.
- Runtime config uses the local OAuth-authenticated Claude Code command (`claude`) instead of model API keys.
- Codex is a setup prerequisite connected to Claude Code plugin/tooling, not a direct server-side runtime command.
- Setup uses `scripts/setup.sh` for POSIX first-run bootstrap and `scripts/setup.mjs` for dependency/setup verification.
- Logging routes through `log()` so later log sinks can be changed in one shared module.
- `dist/` is cleaned before builds so removed source and test modules cannot leave stale compiled output behind.

## Removed Stale Scope

Earlier implementation notes and code had carried forward future automation concepts. Those are no longer active runtime scope:

- No implementer role runtime config.
- No model API key requirement for Claude Code or Codex reviewer execution.
- No label-driven automatic code modification.
- No patch generation/apply loop.
- No loop terminal-state contract.
- No branch-protection verdict check or merge automation.
- No low-risk autonomous readiness policy.

## Red-Green Notes

- A failing phase-scope test first proved that stale runtime modules and exported contracts still existed.
- The Phase 3A green implementation added server runtime metadata while keeping concrete GitHub, git, state, and Claude Code side effects out of the active route layer.
- The Phase 3B green implementation added adapter tests with fake GitHub clients, fake command runners, and temporary SQLite databases before adding concrete adapter modules.
- A review follow-up reproduced the oversized body slow-stream issue with a raw socket test, then changed the server to return `413` with `Connection: close` and stop reading the request immediately after the limit is crossed.
- Review publication metadata was simplified from loop-oriented state to review-only publication state.
- Existing config tests preserve the single environment-read boundary and verify stale runtime env keys are absent.
- A changed-paths regression test now proves empty PR file lists skip before state claim or workspace preparation.
- A git workspace hardening regression test now proves unsafe git arguments are rejected before command execution and isolated git config/env is applied to every workspace command.
- Phase 3C tests first covered replace-env command execution, fail-closed egress guard behavior, secret-scrubbed agent environment construction, guarded Claude Code command construction, PR-checkout cwd separation, structured JSON parsing, sanitized command failures, and app-level `run-independent-reviews` failure publication.
- PR #28 review follow-up added a timeout escalation regression test proving command execution sends `SIGKILL` after the `SIGTERM` grace period when a timed-out child process does not close.
- Phase 3D tests first proved HTTP callbacks needed parsed payloads, long-running dispatch must not block `202` acknowledgements, ordinary PR comments must not create follow-up side effects, GitHub PR reads needed changed-file pagination and metadata parsing, dispatcher mapping failures must skip side effects, and follow-up responder output must remain `analysis-only` and current-SHA-bound.

## Human Editability Rules

- Each source file has one clear reason to exist.
- Domain policy remains pure and does not import adapters.
- App use cases coordinate through explicit ports.
- Harness builders return plain strings and can be edited without understanding SDK clients.
- Runtime execution opens the Phase 3A HTTP server. Phase 3B owns concrete GitHub publication, git workspace, and state adapters. Phase 3C owns Claude Code execution, egress guarding, and agent environment scrubbing. Phase 3D owns recognized-webhook dispatch into app use cases and follow-up responder wiring.

## Setup Usage

- First run on macOS/Linux: `sh scripts/setup.sh`.
- Existing Node 24+ environments: `npm run setup`.
- Start the built runtime with local `.env`: `npm run serve`.
- Run under pm2: `pm2 start ecosystem.config.cjs`.
- The shell bootstrap prefers Volta when available. Without Volta, it installs Node.js under `.tools/node` and does not require `sudo` or a global Node install.
