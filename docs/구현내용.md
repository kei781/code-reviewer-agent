# Phase 0 Implementation Notes

## SDD Intent

Phase 0 is a structure-first implementation. The goal is to make the repository easy for humans and future agents to inspect before any runtime automation is added.

## Source Decisions

- `ADR.md` and `PRD.md` define v5 as a self-hosted review-server flow.
- P0 is review-only. It must not add autofix, formal approval, auto-merge, branch-protection bypasses, or write-token model behavior.
- Claude Code and Codex reviewer passes are represented as independent agent modules with same-level harness builders.
- The orchestrator is modeled as a side-effect-free run plan. Concrete webhook, GitHub, git, sandbox, and model calls remain adapter responsibilities for later phases.

## Phase 0 Deliverables

- Clear TypeScript directory boundaries for `domain`, `app`, `adapters`, `agents`, `orchestration`, `shared`, and `project`.
- A `setup` command that installs npm dependencies, ensures required scaffold directories, and verifies the TypeScript build.
- A POSIX `sh` bootstrap for macOS/Linux first-run setup that uses Volta pins when available, or installs Node.js locally before delegating to the Node setup script.
- A central `log()` helper so application code can route logs through one replaceable function later.
- Tests for phase metadata, directory rules, review-server run-plan construction, and logger routing.
- Documentation that records P0 boundaries and the v5 review-server directory structure.

## Red-Green Notes

- Logger behavior is added test-first: tests assert that a custom sink receives log entries before production logger code exists.
- Review-server run-plan behavior is covered by tests that assert clone, fetch, webhook head-SHA checkout order and independent harness output.
- Existing conflict markers are treated as setup/configuration repair so the test runner can execute meaningful TypeScript tests.

## Human Editability Rules

- Each source file has one clear reason to exist.
- Domain policy remains pure and does not import adapters.
- Harness builders return plain strings and can be edited without understanding SDK clients.
- Runtime execution is represented as data until adapter phases implement concrete side effects.

## Phase 1 Notes

- `RunEnsembleReview` is implemented as a port-driven app use case, not a concrete server or GitHub adapter.
- Unsupported actions, invalid payloads, draft/closed/fork PRs, and already-reviewed SHAs are explicit skip paths.
- Review state is claimed before workspace/orchestrator/publisher side effects, so adapters can implement SQLite delivery/head-SHA idempotency.
- Publication includes reviewed SHA, reviewer agent IDs, kept/dropped/deduped finding counts, merge signal, pass origin, human-review reasons, recommended labels, and review markers.
- Cross-validation keeps only findings with local checkout evidence and corroboration; dropped blocker candidates route to `HUMAN_REVIEW_REQUIRED` rather than `PASS`.
- Required risky paths route to `HUMAN_REVIEW_REQUIRED` and recommend the `security-sensitive` label while still allowing the review pass to run.
- Previously posted finding fingerprints are filtered before publication so synchronize events do not duplicate inline comments.
- Port failures publish structured failure information and return a `failed` result instead of silently rejecting.

## Phase 2 Notes

- `RespondToReviewerMention` implements PRD FR-002 as an app-level use case behind ports.
- Supported `issue_comment` actions are `created` and `edited`.
- The existing pure trigger policy recognizes `@ai-reviewer`, `@claude`, and `/ai review`.
- Ordinary comments, invalid payloads, non-PR issues, closed PRs, and fork PRs are skipped before claim/model/publisher side effects.
- Follow-up state is claimed by delivery, repository, PR number, head SHA, comment ID, and a comment-body revision key before any response generation.
- Duplicate delivery/comment/revision claims skip before response generation, while edited comments with changed bodies can be answered again on the same PR head.
- Concrete issue-comment adapters must enrich raw GitHub payloads with PR metadata such as `headSha` and `isFork` before calling the app use case.
- The response request exposes only `analysis`, `explanation`, `risk-clarification`, and `re-review-signal`; it does not expose fix, approve, merge, push, or write-token actions.
- Fix/merge requests in P0 remain read-only response context. Blocking labels such as `needs-human-review` and `security-sensitive` are carried into the response request so adapters can explain the guard state, while `ai-blocked` hard-skips follow-up work before claim.
- Response generation and publication failures produce structured failure records and mark claimed work as failed.
- If the response comment is already published and state marking fails afterward, the use case records the failure through the state port without publishing a second user-facing failure comment.
- Phase metadata now records Phase 0 and Phase 1 as implemented and Phase 2 as the current implementing phase.

## Phase 3 Notes

- Phase 3 starts the P1 Frontier Pair Autofix Pilot as a policy/contract PR only.
- `src/domain/fixer/actionableMarker.ts` parses exact `<!-- ai-review:actionable ... -->` markers into stable fixer inputs only when the source is a trusted reviewer summary comment; it ignores malformed, untrusted, prefixed, and duplicate marker comments.
- `src/domain/policy/modelPairPolicy.ts` keeps the R/F independence check pure and blocks same model, non-frontier reviewer/fixer, and same model-family pairs.
- `src/domain/policy/autofixPolicy.ts` allows only fresh, same-repo, non-draft, open PRs with `ai-autofix`, unprocessed actionable markers, available fix attempts, no blocking labels, no risky paths, and an independent model pair.
- The Phase 3 decision exposes only `nextAction: "fixer-analyze"` or `nextAction: "skip"`; this PR does not implement patch generation, patch apply, push, approve, merge, auto-merge, write-token fixer behavior, or branch-protection bypass.
- Phase metadata now records Phase 0, Phase 1, and Phase 2 as implemented and Phase 3 as the current implementing phase.

## Phase 4 Notes

- Phase 4 implements pure P1 delta verification and convergence-state contracts.
- `src/domain/convergence/convergenceState.ts` decides `CONVERGED_CLEAN`, `STALLED_OSCILLATING`, `CAPPED_WITH_OPEN`, or `CONVERGING` from typed reviewer/fixer loop facts.
- `CONVERGED_CLEAN` requires the reviewer to have checked the current head SHA, `MERGE_SIGNAL: PASS`, zero unresolved blockers, and no blocker introduced by the fixer diff.
- `CAPPED_WITH_OPEN` recommends `needs-human-review` when fix attempts reach the configured cap while blockers remain.
- `STALLED_OSCILLATING` recommends `needs-human-review` when blocker counts stop decreasing, blocker classes repeat, or the fixer diff introduces a blocker.
- `src/domain/review/orchestratorStateMarker.ts` parses hidden `ai-orchestrator` comment markers for epoch, reviewed SHA, fixer SHA, fix attempts, processed ids, blocker history, and fixer run id.
- Phase 4 does not implement reviewer execution, fixer patch application, GitHub comment updates, status checks, auto-merge, or write-token behavior.
- Phase metadata now records Phase 0 through Phase 3 as implemented and Phase 4 as the current implementing phase.
